## North star

> Cycle 24 taught the Librarian what NOT to do (EPM — learn from rejections). Cycle 25
> teaches it what TO do (CTM — learn from approvals). On every approval, kage records
> the successful decision as a precedent note. At the next distillation, the Librarian
> retrieves recent precedents and injects them as few-shot positive examples, biasing
> toward PROMOTE patterns that match past approved decisions. This closes the dual-memory
> loop from MemAPO (arxiv 2603.21520): EPM stops recurring mistakes, CTM reinforces
> correct patterns.

Scope: **one agent (Librarian), one signal (approve), recency-based runtime injection.**

---

## Why runtime, not batch

EPM (Cycle 24) works as batch because rejection patterns are stable: "don't promote stale
scout findings" applies regardless of current content. CTM is different. Approval precedents
are **content-adjacent**: a past approval for a "Python type-hints note" is more useful when
evaluating another code snippet than when evaluating a calendar event. Batch-distilling
approvals into generic rules destroys specificity — the signal degrades to vague encouragement.

The right mechanism: retrieve relevant precedents at distill time, inject only confirmed
examples. v1 uses recency as the retrieval signal (simplest, lowest risk); v2 adds BM25
relevance ranking; v3 adds a Qwen3 LLM-check gate when ranking quality needs improvement.

## What changed from v1 → v2 (✎CR1)

1. ✎CR1 **BLOCKER — `_emit_ctm_note` signature fixed.** v1 passed `reason_from_aq` from
   `write_note` caller scope, but that variable does not exist there. v2: `_emit_ctm_note`
   takes only `(approval_id, home)` and does all its own SELECTs. `note.get("action")`
   was also wrong — `action` lives in `approval_queue.action`, not in `note_json`.

2. ✎CR1 **BLOCKER — CTM notes stored with `project='kage-ctm-librarian'`** instead of
   `project IS NULL`. v1's `WHERE m.project IS NULL` would contaminate results with any
   user note that happened to contain the words "CTM" and "log". Fix: full project tag,
   same pattern as EPM (`kage-corrections-librarian`), added to `_ALWAYS_LOCAL_PROJECTS`.
   `_retrieve_ctm` SELECT filters on `m.project = 'kage-ctm-librarian'`.

3. ✎CR1 **BLOCKER — PII re-gate before cloud injection.** `ctm_text` (title + source +
   reason generated by the cloud) can embed PII that slipped `_gate_text`. Fix: call
   `_gate_text(ctm_text, cfg)` before appending to `results`.

4. ✎CR1 **REQUIRED — latency cap dropped.** v1 proposed 50 Qwen3 calls per invocation
   (~25s worst case). v2 drops Qwen3 from the retrieval path entirely for now. See below.

5. ✎CR1 **REQUIRED — switch v1 retrieval to recency, justify Qwen3 deferral.** Qwen3
   LLM-check is the v3 upgrade path. v1 uses `ORDER BY created_at DESC LIMIT max_examples`
   — O(1) DB query, no Ollama call. `ponytail:` names the ceiling and the upgrade path.

6. ✎CR1 **REQUIRED — three missing tests added to test plan** (see §Test plan).

7. ✎CR1 **REQUIRED — `state='baseline'` specified** for CTM note INSERT. CTM notes are
   internal kage system notes, not user project notes — they use `'baseline'` intentionally
   despite having a non-NULL project (the `project` field names the CTM store, not a user
   project). Note: this is the OPPOSITE of `write_note`'s convention (which uses `'scoped'`
   for non-NULL project), because those are user memory notes, not internal system records.

## What changed from v2 → v3 (✎CR2)

1. ✎CR2 **REQUIRED — `needs_embed=0` added to INSERT.** v2's INSERT omitted `needs_embed`,
   so the schema default of 1 applied — CTM notes would be swept into ChromaDB on
   `kage reindex`, contradicting the pitch's own "no ChromaDB" architecture claim. Fix:
   `needs_embed=0` explicit in INSERT VALUES. Not a privacy issue (Ollama is local) but
   a correctness violation of the spec.

2. ✎CR2 **REQUIRED — triple-connection collapsed to single JOIN.** v2 opened three
   separate SQLite connections (conn for the action/note JOIN, conn2 for reason, conn3
   for INSERT). The comment `"reason column added in Cycle 24 approval_queue migration"`
   was factually wrong — `reason TEXT NOT NULL` is in the original `CREATE TABLE IF NOT
   EXISTS approval_queue`, not an ALTER TABLE migration. Fix: single connection, full
   JOIN including `aq.reason` in one SELECT, reuse connection for INSERT.

3. ✎CR2 **REQUIRED — added test C-bis (OSError path in `_retrieve_ctm`).** The
   `except OSError: continue` branch was untested. New test: write a CTM DB row but NOT
   the markdown file → assert `_retrieve_ctm` returns `[]` without raising.

4. ✎CR2 **REQUIRED — Test E setup specified explicitly.** v2 left the PII injection
   mechanism underspecified. v3 specifies: insert a CTM note file directly with an email
   address in `ctm_body`, call `_retrieve_ctm`, assert `[REDACTED_PII]` in result. No
   need to go through `_emit_ctm_note`; direct file write is simpler and tests `_retrieve_ctm`
   in isolation.

## Architecture (single path)

```
  kage librarian approve <id>
        │
        ▼
  write_note(approval_id): write approved memory → permanent storage [existing]
        │  success = True
        │
        ├── Step 7: UPDATE approval_queue SET decision='approved'
        │
        ▼
  _emit_ctm_note(approval_id, home)           [NEW — best-effort, wrapped in try/except]
        │  SELECT action, reason, note_json FROM approval_queue WHERE id=?
        │  Parse title from note_json; get source from staging row
        │  body = "CTM log — Librarian approval: {title}. Source: {source}.
        │          Action: {action}. Reason: {reason or 'approved'}."
        │  INSERT into memories (project='kage-ctm-librarian', state='baseline')
        │  INSERT into memory_identities, memory_projects, memory_fts
        │  Write markdown file to home/memory/{note_id}.md
        │  (no chunks, no ChromaDB — CTM notes are short metadata, not embedded)
        ▼
  kage-ctm-librarian  [_ALWAYS_LOCAL_PROJECTS, project-filtered SELECT]


  distill_and_judge(content, source)
        │
        ▼
  Step C2 — _retrieve_ctm(content[:400], cfg, home, max_examples=3)
        │  SELECT recent CTM notes WHERE project='kage-ctm-librarian'
        │    ORDER BY created_at DESC LIMIT max_examples
        │  For each: read file → _gate_text(ctm_text, cfg) → append to results
        │  Returns [] on any exception (graceful fallback)
        │
        ▼
  System prompt assembly (order — last prepend appears topmost):
        Step D1: EPM prepend (if epm_enabled + learned rules exist):
          system = "[kage learned librarian rules]\n{rules}\n\n{_DISTILL_SYSTEM}"
        Step D2: CTM prepend (if ctm_enabled + examples returned):
          system = "[kage CTM precedents]\n{examples}\n\n{system}"
        Final order in system: CTM → EPM → _DISTILL_SYSTEM
        │
        ▼
  runtime.cloud.complete(provider, system, messages, cfg)
```

CTM appears before EPM because precedents (specific positive examples) should frame the
task before the rules (general negative constraints) impose guardrails.

## Privacy design

`kage-ctm-librarian` notes contain only metadata:
- `title` — cloud-generated 5–8 word summary (sanitized by `_gate_text` before distill)
- `source` — "scout"/"user"/"librarian" (enum, not sensitive)
- `action` — PROMOTE/HOLD/DISCARD
- `reason` — cloud-generated one-sentence justification (may embed content fingerprints)

**✎CR1 fix:** before injecting `ctm_text` into cloud system prompt, run `_gate_text`
on it. Title and reason pass through PII regex scrubbing before cloud sees them.

`_ALWAYS_LOCAL_PROJECTS` prevents CTM notes from surfacing via `_disclosure_gate` or
`_gate_conversation` (the recall/ask/chat paths). `_retrieve_ctm` bypasses both gates
intentionally — it reads locally for local-to-cloud injection, and the PII re-gate
substitutes for the normal gate function.

**Cross-project:** CTM notes do NOT carry the original approved note's project. The
`project` field in `memories` is `'kage-ctm-librarian'` (the CTM store name), not the
project of the approved content. Precedents are personal-identity scoped, not
project-scoped. If practice shows project scoping is needed, add a `ctm_project` column
in a later cycle.

```python
def _emit_ctm_note(approval_id: str, home: pathlib.Path) -> None:
    """Write a CTM precedent note for a successful Librarian approval.
    Self-contained: single connection, all SELECTs in one JOIN."""
    # ✎CR2: single connection (was three); aq.reason is in original CREATE TABLE, not a migration.
    conn = _connect()
    try:
        row = conn.execute(
            "SELECT aq.action, aq.note_json, aq.reason, sq.source "
            "FROM approval_queue aq "
            "LEFT JOIN staging_queue sq ON sq.id = aq.staging_id "
            "WHERE aq.id = ?",
            (approval_id,),
        ).fetchone()
        if not row:
            return
        action = row[0] or "promote"
        try:
            note_meta = json.loads(row[1]) if row[1] else {}
        except json.JSONDecodeError:
            note_meta = {}
        title = note_meta.get("title", "")
        reason = row[2] or ""
        source = row[3] or ""

        ctm_body = (
            f"CTM log — Librarian approval: {title}. "
            f"Source: {source}. Action: {action}. Reason: {reason or 'approved'}."
        )
        note_id = str(uuid.uuid4())
        rel_path = f"memory/{note_id}.md"
        ts = datetime.now().astimezone().isoformat(timespec="seconds")

        conn.execute(
            "INSERT INTO memories"
            " (id, content_path, project, created_at, local_only, needs_embed, state)"
            " VALUES (?, ?, ?, ?, 0, 0, ?)",
            # ✎CR2: needs_embed=0 — CTM notes are short metadata, not for ChromaDB.
            (note_id, rel_path, "kage-ctm-librarian", ts, "baseline"),
        )
        conn.execute(
            "INSERT OR IGNORE INTO memory_identities(mem_id, identity) VALUES (?, ?)",
            (note_id, "personal"),
        )
        conn.execute(
            "INSERT OR IGNORE INTO memory_projects(mem_id, project) VALUES (?, ?)",
            (note_id, "kage-ctm-librarian"),
        )
        conn.execute(
            "INSERT INTO memory_fts (id, body) VALUES (?, ?)",
            (note_id, ctm_body),
        )
        conn.commit()
    finally:
        conn.close()

    full_path = home / rel_path
    full_path.parent.mkdir(parents=True, exist_ok=True)
    full_path.write_text(
        f"---\nid: {note_id}\nproject: kage-ctm-librarian\n"
        f"created_at: {ts}\nsource: librarian-approve\nidentities:\n"
        f"  - personal\nstate: baseline\n---\n\n{ctm_body}\n"
    )
```

### Step 3 — `_retrieve_ctm` (librarian.py)

```python
def _retrieve_ctm(
    content_preview: str,
    cfg: dict,
    home: pathlib.Path,
    max_examples: int = 3,
) -> list[str]:
    """Return up to max_examples recent CTM precedent strings (PII-gated for cloud).
    ponytail: recency-based retrieval — O(1) DB query, no Ollama calls.
    Ceiling: accuracy degrades when CTM store grows large and recency ≠ relevance.
    Upgrade path v2: add FTS5 BM25 relevance ranking (rank by content_preview words).
    Upgrade path v3: add Qwen3 LLM-check gate (Y/N per candidate, 50-note scan cap)."""
    try:
        db_path = home / "indexes" / "kage.db"
        conn = sqlite3.connect(str(db_path))
        try:
            rows = conn.execute(
                "SELECT id, content_path FROM memories"
                " WHERE project = 'kage-ctm-librarian'"
                " ORDER BY created_at DESC LIMIT ?",
                (max_examples,),
            ).fetchall()
        finally:
            conn.close()

        results: list[str] = []
        for note_id, rel_path in rows:
            try:
                ctm_text = (home / rel_path).read_text()
                ctm_text = _gate_text(ctm_text, cfg)  # ✎CR1: PII pass before cloud
                results.append(ctm_text.strip())
            except OSError:
                continue
        return results
    except Exception:
        return []
```

## What this is NOT doing (deferred)

- **BM25 relevance ranking (v2):** v1 uses recency. When CTM store exceeds ~20 notes,
  recency and relevance diverge. v2 fix: rank CTM notes by FTS5 BM25 score against
  `content_preview` words.
- **Qwen3 LLM-check gate (v3):** highest-quality retrieval but ~N×500ms latency. Add
  when BM25 ranking quality is insufficient. Seam: replace the DB ORDER BY clause with
  a Qwen3-filtered loop capped at 10 candidates and 5s wall-clock.
- **Project-scoped CTM (v2):** cross-project precedents allowed in v1. Add
  `WHERE m.ctm_source_project = ?` filter if practice shows cross-project bleeding.
- **CTM for Scout/Monitor:** one agent first. Seam: `_emit_ctm_note` is parameterized
  by `approval_id`; other agents can call it with their own approval queue rows.
- **`kage learn --ctm` command:** no batch step needed — CTM is runtime retrieval.
